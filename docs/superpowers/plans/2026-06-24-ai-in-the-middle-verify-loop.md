# AI-in-the-middle verify loop — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Reintroduce a Claude cleanup+chord step between Audiveris export and EasyABC in the per-tune verify loop, while preserving the live auto-refreshing compare and the queue/loop machinery.

**Architecture:** Two small code changes to existing Bash orchestration, plus an SOP write-up. (1) Extract the inline live-compare watcher in `verify_tune.sh` into a reusable `live_compare.sh` helper, and rewire the existing `--no-export` manual path to sit on top of it (behavior-preserving refactor). (2) Add an `--export-only` flag that runs Audiveris + export and then stops (no title fix, no EasyABC), giving Claude a clean seam to do stage-2 cleanup. Claude itself orchestrates the 3-stage loop interactively using these pieces; that orchestration is documented as the SOP, not coded.

**Tech Stack:** Bash, flatpak Audiveris, EasyABC, Firefox, ImageMagick (`convert`), existing pipeline scripts (`export_tune.sh`, `render_abc.sh`, `make_compare.sh`, `promote_tune.sh`). Linux/Mint, X11 (`wmctrl`).

### Task 1: Extract `live_compare.sh` reusable watcher

**Files:**
- Create: `notation_pipeline/bin/live_compare.sh`

**Interfaces:**
- Produces: `live_compare.sh <candidate_abc> <render_png> <scan_png>` — builds an initial side-by-side compare, writes an auto-refreshing Firefox page, opens it, then polls the candidate ABC's mtime and rebuilds the compare on every save. - [ ] **Step 1: Write the helper**

Create `notation_pipeline/bin/live_compare.sh` with exactly this content (it reproduces the inline `build_compare` + watcher + HTML page from `verify_tune.sh`, parameterized):

```bash
#!/usr/bin/env bash
# Live auto-refreshing side-by-side compare for the verify loop.
#
# Builds a source-vs-render compare PNG, writes a self-refreshing HTML page that
# reloads the PNG every 1.5s (so external EasyABC saves show up without piling
# new Firefox tabs), opens it in Firefox, then polls the candidate ABC's mtime
# and rebuilds the compare on every save. Blocks until killed (the caller owns
# the lifecycle — e.g. kills it when EasyABC closes).
#
# Usage: live_compare.sh <candidate_abc> <render_png> <scan_png>
#
# Linux/Mint-bound (firefox, ImageMagick via the pipeline helpers).
set -uo pipefail

HERE="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

CAND_ABC="${1:?usage: live_compare.sh <candidate_abc> <render_png> <scan_png>}"
RENDER="${2:?missing render png path}"
SCAN="${3:?missing scan png path}"

SLUG="$(basename "$CAND_ABC" | tr -c 'A-Za-z0-9' '_')"
COMPARE_PNG="/tmp/${SLUG}.compare.png"
COMPARE_HTML="/tmp/${SLUG}.compare.html"

build_compare() {  # (re)build the side-by-side compare from the current render
    bash "${HERE}/render_abc.sh" "$CAND_ABC" "$RENDER" >/dev/null 2>&1 || return 1
    bash "${HERE}/make_compare.sh" "$SCAN" "$RENDER" "$COMPARE_PNG" >/dev/null 2>&1 || return 1
}

build_compare || echo "  (initial compare build had a hiccup — continuing)"

cat > "$COMPARE_HTML" <<HTML
<!doctype html><html lang="en"><head><meta charset="utf-8">
<title>$(basename "$CAND_ABC") — compare</title>
<style>body{margin:0;background:#222;text-align:center}
img{display:inline-block;max-width:100%;height:auto}</style>
</head><body>
<img id="c" src="${SLUG}.compare.png" alt="compare">
<script>
setInterval(function(){
  document.getElementById('c').src='${SLUG}.compare.png?t='+Date.now();
}, 1500);
</script></body></html>
HTML
firefox "$COMPARE_HTML" >/dev/null 2>&1 &

# Rebuild on every save (mtime poll — no inotify dependency).
last=$(stat -c %Y "$CAND_ABC" 2>/dev/null || echo 0)
while sleep 1; do
    now=$(stat -c %Y "$CAND_ABC" 2>/dev/null || echo 0)
    if [[ "$now" != "$last" ]]; then
        last="$now"
        build_compare
    fi
done
```

1. **Stage 1 — `verify_tune.sh --export-only "$TUNE"`**: human corrects OMR in Audiveris + closes; export produces `abc/<Tune>-candidate.abc`.
2. **Stage 2 — Claude cleanup**: read `source_images/<Tune>.png` + the candidate ABC; fix header (`T:` title, `C:` attribution, `M:` meter incl. cut-time `M:C|`, `K:` key, `L:`), structure (linebreaks `$` to match scan systems, repeats `|: :|`, return/volta endings with `%%contbarnb 1`, slurs), OMR error patterns (octave shifts, voice-residue `[xy]` chords, two-voice splits), and best-effort chords (capital roots, beat placement, spacing). 3. **Stage 3 — live review**: Claude runs `live_compare.sh "$CAND" "$REN" "$SCAN" &` in the background, launches `easyabc "$CAND"` tracking its PID. KILL/REOPEN RULE: any Claude edit to the candidate while EasyABC is open → kill EasyABC, edit, relaunch (new PID). Claude tracks whether it initiated a kill: if EasyABC exits and Claude did NOT kill it, the human closed it → run the promote gate (`promote_tune.sh` + guarded `git add && … && commit`).
