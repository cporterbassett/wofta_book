Root cause: The 2× Lanczos upscale doubles the pixel size of key-signature sharp symbols. Audiveris's key-sig classifier appears calibrated for a specific size range — at 2× the sharps fall outside it. Tunes with a smaller original staff (Arkansas Traveler ~154 DPI, Ashokan Farewell ~152 DPI) fail; tunes with a larger staff (Cincinnati Hornpipe ~168 DPI, Blackberry Blossom ~164 DPI) still pass. Blanking chord names made things worse, not better — chord names were not the cause. When key detection fails, all note pitches dependent on that key become wrong (e.g. F is written as natural instead of F♯ in D major).

B,C|:DFED B,2B,2|A,2A,2 D4| E2E2 F2F2|DFED B,2A,2| ... |:agfa gfeg|fedf edce|...

Rendered

Different arrangement from the WOFTA version — pickup is B,C (low B then middle C) vs. WOFTA's pickup starting on D. Ending phrases also differ slightly. Useful as a pitch-accuracy reference but not a drop-in gold standard.

Diagnosis

The key signature (2 sharps for D major) is clearly visible in the image — Audiveris is failing to recognize it, not failing to find it. The MXL measure-1 <attributes> block contains <time> (4/4) but no <key> element at all; the key only appears at line 300 of the XML as a mid-tune change, which xml2abc ignores for the global K: header.

  • Why the 2× upscale breaks it: at the original resolution, the sharp symbols are ~13 px apart (staff interline spacing). At 2×, they're ~26 px. Audiveris's key-sig classifier appears to have a narrower expected size range — the symbols at 2× no longer match.
  • Why blanking didn't help: chord names aren't touching the key-sig region. The 2× resize was the culprit all along.
  • Fix: Use the original image (or unsharp-only preprocessing) for Audiveris. The original gives correct K:D and correct note pitches.

Diagnosis — Hardest case

Ashokan Farewell fails even on the original image. Two compounding factors:

  • Lowest DPI in the set (~152). At original resolution, the sharp symbols and time-sig numerals are smaller than in the other tunes. Audiveris finds one sharp (G major) but misses the second (D major).
  • Pickup measure. The tune starts with a partial measure (anacrusis) before the first full bar. This unusual first-measure structure may cause Audiveris to fail its time-signature search, which typically looks for a complete beat pattern in measure 1.
  • Copyrighted tune. No thesession.org reference is available — the gold standard must be transcribed entirely from the WOFTA image.
  • Recommended path: Try the Audiveris GUI (flatpak run org.audiveris.audiveris) on the original image — the interactive overlay lets you see what was recognised and manually correct the key/time sig before exporting.

thesession.org #1365 — note the different meter

M:4/4 L:1/8 K:Gmaj ge|dBAc BGGA|B2BA (3Bcd ge| dBAc BG~G2|eaag ageg|...

Rendered

thesession.org uses M:4/4 while the WOFTA version is in cut time (M:2/2). The melody is the same (same number of notes per measure) — cut time is just a different notation convention for the same rhythm. Also: the B section in the WOFTA version goes to Em/B7, while this thesession version stays in G throughout — these are genuinely different arrangements.

Why this one works (and thesession can't be used directly)

At ~164 DPI, Blackberry Blossom's staff is large enough that the single sharp (G major) lands in Audiveris's recognisable size range even after 2×. Cut time (2/2 alla breve) is also apparently more reliably detected than 3/4.

  • The B section differs: WOFTA has Em and B7 chord sections; thesession.org stays in G throughout. The gold standard must come from the WOFTA image, not thesession.
  • Scan quality: Despite being a lower-quality photocopy, the preprocessing handled it well — the sharpening step compensates for the slight blurriness.

Key Insight: Staff Size Predicts Key-Detection Success

Tune Orig. DPI (est.) Staff interline at 2× K detected at 2×?
Ashokan Farewell ~152 DPI ~26 px No
Arkansas Traveler ~154 DPI ~26 px No
Blackberry Blossom ~164 DPI ~28 px Yes
Cincinnati Hornpipe ~168 DPI ~30 px Yes

The threshold appears to be around 28–30 px staff interline spacing at 2× scale (~14–15 px at original). The WOFTA scans vary in resolution — some will hit this threshold, others won't.

Next step to test: Run Angeline the Baker (which achieved 100% with 2×+unsharp) through the unsharp-only pipeline to see whether note accuracy suffers. If unsharp-only is close to 100%, it's the safer default for all 277 tunes — it preserves key detection across the full DPI range.
